# Hadoop এবং HDFS আর্কিটেকচার — সম্পূর্ণ বাংলা গাইড

> **মূল দর্শন:** "ডেটাকে কম্পিউটেশনের কাছে নিয়ে যাওয়ার বদলে, কম্পিউটেশন (কোড)-কেই ডেটার কাছে পাঠাও।"

---

## Hadoop কী এবং কেন দরকার?

একটি সাধারণ সার্ভারের পক্ষে ১ টেরাবাইট ডেটা প্রসেস করতে ঘণ্টার পর ঘণ্টা লেগে যায়। কিন্তু যদি একই কাজ ১০০টি সার্ভার মিলে করে, তাহলে সময় ১০০ গুণ কমে যাওয়ার কথা। Hadoop ঠিক এই কাজটিই করে।

Hadoop মূলত দুটো বড় সমস্যা সমাধান করে:
1. **ডেটা কোথায় রাখা হবে?** → এটা সমাধান করে **HDFS**
2. **ডেটা কীভাবে প্রসেস করা হবে?** → এটা সমাধান করে **MapReduce/Spark**, এবং রিসোর্স ম্যানেজ করে **YARN**

---

## Hadoop-এর তিনটি মূল স্তর (Layers)

```
┌─────────────────────────────────────────────┐
│        Processing Layer (MapReduce/Spark)    │  ← ডেটা প্রসেসিং
├─────────────────────────────────────────────┤
│       Resource Layer (YARN)                  │  ← CPU/RAM ম্যানেজমেন্ট
├─────────────────────────────────────────────┤
│       Storage Layer (HDFS)                   │  ← ডেটা স্টোরেজ
└─────────────────────────────────────────────┘
```

## ১. HDFS — Hadoop Distributed File System

HDFS হলো Hadoop-এর স্টোরেজ ইঞ্জিন। এটি একটি **Fault-tolerant, Distributed** ফাইল সিস্টেম যা হাজারো মেশিনের হার্ড ড্রাইভকে একটি একক, বিশাল ড্রাইভের মতো ব্যবহার করতে দেয়।

### HDFS-এর মূল উপাদান

---

### ১.১ NameNode — ক্লাস্টারের মস্তিষ্ক

NameNode হলো HDFS-এর **Master**। এটি আসল ডেটা স্টোর করে না, বরং ডেটার সম্পর্কে তথ্য (Metadata) ধরে রাখে।

**NameNode কী কী Metadata রাখে?**
- ফাইলের নাম, পাথ, সাইজ এবং পারমিশন
- কোন ফাইল কতগুলো ব্লকে বিভক্ত
- প্রতিটি ব্লক কোন কোন DataNode-এ আছে

**NameNode কীভাবে ডেটা সেভ করে?**

NameNode দুটি ক্রিটিক্যাল ফাইল ব্যবহার করে:

- **`FSImage` (File System Image):** পুরো ফাইল সিস্টেমের একটি সম্পূর্ণ স্ন্যাপশট। এটি অনেকটা একটি ডেটাবেসের ফুল ব্যাকআপের মতো।
- **`EditLog`:** NameNode চালু হওয়ার পর থেকে যত পরিবর্তন হয়েছে (ফাইল যোগ, ডিলিট, রিনেম) সব কিছুর একটি লগ। অনেকটা ব্যাংকের ট্রানজেকশন লগের মতো।

NameNode চালু হলে প্রথমে `FSImage` লোড করে, তারপর `EditLog` প্রয়োগ করে সিস্টেমের সর্বশেষ অবস্থায় ফিরে আসে।

> **গুরুত্বপূর্ণ:** NameNode তার সমস্ত Metadata **RAM-এ** রাখে, তাই এটি অত্যন্ত দ্রুত। কিন্তু এই কারণেই বড় ক্লাস্টারে NameNode সার্ভারে অনেক বেশি RAM থাকা জরুরি।

---

### ১.২ Secondary NameNode — Checkpointing-এর সহকারী

> ⚠️ **সবচেয়ে প্রচলিত ভুল ধারণা:** Secondary NameNode মানেই NameNode-এর "হট ব্যাকআপ" নয়! NameNode ডাউন হলে এটি তার জায়গা নেয় না।

**Secondary NameNode-এর আসল কাজ হলো Checkpointing:**

সময়ের সাথে সাথে `EditLog` ফাইলটি অনেক বড় হয়ে যায়। যদি NameNode রিস্টার্ট হয়, তাহলে সেই বিশাল EditLog প্রসেস করতে অনেক সময় লাগে।

Secondary NameNode এই সমস্যার সমাধান করে:
1. নির্দিষ্ট সময় অন্তর (ডিফল্ট: ১ ঘণ্টা) সে NameNode থেকে `FSImage` এবং `EditLog` নিয়ে আসে।
2. দুটো ফাইল মার্জ (Merge) করে একটি নতুন, আপডেটেড `FSImage` তৈরি করে।
3. সেই নতুন `FSImage` NameNode-কে পাঠিয়ে দেয় এবং `EditLog` রিসেট হয়।

এই প্রক্রিয়াকে বলে **Checkpointing।**

### ১.৩ DataNode — ডেটার আসল ঘর

DataNode হলো HDFS-এর **Slave/Worker**। এগুলো সেই সব মেশিন যেখানে আসল ডেটার ব্লকগুলো হার্ড ড্রাইভে সেভ থাকে। একটি প্রোডাকশন ক্লাস্টারে শত শত বা হাজারো DataNode থাকতে পারে।

**DataNode-এর প্রতিদিনের কাজ:**

- **Block Storage:** ফাইলের ব্লকগুলো লোকাল ডিস্কে সেভ করা এবং ক্লায়েন্টকে রিড/রাইট করতে দেওয়া।
- **Heartbeat:** প্রতি **৩ সেকেন্ড** অন্তর NameNode-কে একটি ছোট সিগন্যাল পাঠানো — "আমি সচল আছি।" যদি NameNode পর পর **১০টি Heartbeat** না পায় (মোট ৩০ সেকেন্ড), তাহলে সে ধরে নেয় DataNode-টি মারা গেছে।
- **Block Report:** প্রতি **৬ ঘণ্টা** অন্তর সম্পূর্ণ Block লিস্ট NameNode-কে পাঠানো।

**DataNode ডাউন হলে কী হয়?**

NameNode তখন সেই DataNode-এর সব ব্লকের ডেটা অন্য সচল DataNode-এ নতুন করে কপি করা শুরু করে, যাতে Replication Factor সবসময় ঠিক থাকে।

---

### ১.৪ Block এবং Replication

**Block কী?**

HDFS-এ কোনো ফাইল সরাসরি সেভ হয় না। ফাইলটিকে প্রথমে সমান সমান টুকরোয় (Block) ভাগ করা হয়। ডিফল্ট সাইজ **১২৮ MB**।

উদাহরণ: একটি ৫০০ MB ফাইল `4টি ব্লকে` ভাগ হবে:
- Block-1: ১২৮ MB
- Block-2: ১২৮ MB
- Block-3: ১২৮ MB
- Block-4: ১১৬ MB (বাকি অংশ)

**Replication কী?**

প্রতিটি ব্লকের **৩টি কপি** (ডিফল্ট) আলাদা আলাদা DataNode-এ সেভ থাকে। এর মানে একটি মেশিনের হার্ড ড্রাইভ পুড়ে গেলেও ডেটা নিরাপদ।

**Rack Awareness — বুদ্ধিমান রিপ্লিকেশন:**

একটি বড় ডেটা সেন্টারে অনেক মেশিন একটি র‍্যাকের (Rack) মধ্যে থাকে এবং একটি সুইচের সাথে যুক্ত থাকে। যদি সেই সুইচটি নষ্ট হয়, পুরো র‍্যাকই অফলাইন হয়ে যাবে।

Hadoop এই কারণে বুদ্ধিমানভাবে রিপ্লিকেশন করে:
- **১ম কপি:** যে মেশিন থেকে রাইট হচ্ছে, সেখানেই অথবা একই র‍্যাকে।
- **২য় কপি:** একটি **ভিন্ন র‍্যাকে** অন্য মেশিনে।
- **৩য় কপি:** সেই ভিন্ন র‍্যাকেই আরেকটি মেশিনে।

এভাবে, পুরো একটি র‍্যাক অফলাইন হলেও ডেটার অন্তত একটি কপি সবসময় পাওয়া যাবে।

### ১.৫ HDFS Read/Write প্রক্রিয়া

**ডেটা লেখার (Write) ধাপসমূহ:**

```
Client                NameNode              DataNode-1  DataNode-2  DataNode-3
  |                      |                      |           |           |
  |--- লেখার অনুরোধ --->|                      |           |           |
  |<-- DN List দেয় ------|                      |           |           |
  |                      |                      |           |           |
  |---------- ব্লক পাঠাও (Pipeline) ----------->|           |           |
  |                      |              DN-1 -->|           |           |
  |                      |                      |-- DN-2 -->|           |
  |                      |                      |           |-- DN-3 -->|
  |                      |                      |<-- OK ----|           |
  |                      |               <-- OK-|           |           |
  |<-- লেখা সফল ---------|                      |           |           |
```

1. ক্লায়েন্ট NameNode-কে বলে: "আমি একটি ফাইল লিখতে চাই।"
2. NameNode ব্লকগুলো কোথায় রাখতে হবে তার একটি DataNode-এর তালিকা দেয়।
3. ক্লায়েন্ট প্রথম DataNode-কে একটি ব্লক পাঠায়। সেই DataNode সেটি পাওয়ার সাথে সাথে দ্বিতীয় DataNode-কে পাঠায়, এবং এভাবে **Pipeline** তৈরি হয়।
4. সব DataNode সফলভাবে পেলে একটি **Acknowledgment** ক্লায়েন্টের কাছে ফিরে আসে।

**ডেটা পড়ার (Read) ধাপসমূহ:**

1. ক্লায়েন্ট NameNode-কে জিজ্ঞেস করে: "`/data/flights.csv` ফাইলটা কোথায় আছে?"
2. NameNode সেই ফাইলের ব্লকগুলোর অবস্থান এবং DataNode-এর লিস্ট দেয় (সবচেয়ে কাছের/দ্রুত ডেটা-নোড আগে)।
3. ক্লায়েন্ট সরাসরি সেই DataNode-গুলো থেকে ব্লক পড়ে নেয়। NameNode আর মাঝখানে থাকে না।

> **মূল পয়েন্ট:** রিড করার সময় ডেটা NameNode-এর মধ্য দিয়ে আসে না। NameNode শুধু ম্যাপ দেয়, বাকি কাজ Client এবং DataNode সরাসরি করে।

---

### ১.৬ Data Integrity — ডেটা কি ঠিকঠাক আছে?

ডেটা লেখার সময় HDFS প্রতিটি ব্লকের একটি **Checksum** (MD5/CRC32) তৈরি করে। যখন ডেটা পড়া হয়, তখন সেই Checksum আবার মিলিয়ে দেখা হয়। যদি না মেলে, HDFS বুঝতে পারে ব্লকটি করাপ্টেড। তখন সে অন্য DataNode থেকে সুস্থ কপিটি দিয়ে দেয়।

## ২. YARN — Yet Another Resource Negotiator

YARN হলো Hadoop-এর রিসোর্স ম্যানেজার। এটি ঠিক করে কোন অ্যাপ্লিকেশন কতটুকু CPU এবং RAM পাবে।

### YARN-এর উপাদানসমূহ

**ResourceManager (RM):**
- পুরো ক্লাস্টারের একটিমাত্র মাস্টার।
- দুটি অংশ আছে:
  - **Scheduler:** রিসোর্স ভাগ করে দেওয়ার নীতি ঠিক করে।
  - **ApplicationsManager:** জমা পড়া কাজের (Job) তত্ত্বাবধান করে।

**NodeManager (NM):**
- প্রতিটি মেশিনে একটি করে থাকে।
- সেই মেশিনের CPU এবং RAM-এর ব্যবহার পর্যবেক্ষণ করে এবং ResourceManager-কে জানায়।

**ApplicationMaster (AM):**
- যখন আপনি একটি Spark বা MapReduce জব সাবমিট করেন, তখন সেই জবের জন্য একটি **নতুন ApplicationMaster** তৈরি হয়।
- AM-এর কাজ হলো তার নিজের জবের জন্য ResourceManager-এর কাছ থেকে রিসোর্স (Container) চাওয়া এবং NodeManager-এর সাথে কাজ সম্পন্ন করানো।
- জব শেষ হলে AM মারা যায়।

**Container:**
- এটি একটি নির্দিষ্ট পরিমাণ RAM এবং CPU-র বান্ডেল।
- উদাহরণ: "Container = ২ CPU cores + ৪ GB RAM"।
- সমস্ত Map Task এবং Reduce Task এই Container-এর ভেতরেই চলে।

### YARN-এর Scheduling নীতি

| Scheduler | কীভাবে কাজ করে |
|---|---|
| **FIFO** | প্রথমে আসলে প্রথমে সার্ভিস। সহজ কিন্তু বড় জব ছোট জবকে আটকে রাখতে পারে। |
| **Capacity** | বিভিন্ন টিম বা বিভাগের জন্য আলাদা আলাদা ক্যাপাসিটি ভাগ করে দেওয়া। |
| **Fair** | সব চলমান জবের মধ্যে রিসোর্স সমানভাবে ভাগ হয়। কেউ বঞ্চিত হয় না। |

## ৩. MapReduce — ডেটা প্রসেসিং মডেল

MapReduce হলো Hadoop-এ ডেটা প্রসেস করার মূল পদ্ধতি। এর তিনটি ধাপ:

### ধাপ ১: Map Phase

ইনপুট ডেটাকে ছোট ছোট চাঙ্কে ভাগ করে প্রতিটি চাঙ্কে Map ফাংশন চালানো হয়। এটি প্রতিটি রেকর্ড থেকে একটি **(Key, Value)** জোড়া তৈরি করে।

উদাহরণ (Word Count):
- ইনপুট: `"Hadoop is great, Spark is great"`
- আউটপুট: `(Hadoop,1) (is,1) (great,1) (Spark,1) (is,1) (great,1)`

### ধাপ ২: Shuffle and Sort — সবচেয়ে গুরুত্বপূর্ণ ধাপ

এই ধাপে একই Key-এর সমস্ত Value-কে একত্রিত করা হয়। এটি নেটওয়ার্কের মাধ্যমে বিভিন্ন মেশিন থেকে ডেটা এক জায়গায় আনে।

- আউটপুট: `(great,[1,1]) (Hadoop,[1]) (is,[1,1]) (Spark,[1])`

### ধাপ ৩: Reduce Phase

প্রতিটি Key-এর Value-গুলো যোগ করে ফাইনাল আউটপুট তৈরি করা হয়।

- ফাইনাল আউটপুট: `(great,2) (Hadoop,1) (is,2) (Spark,1)`

> **আধুনিক বিকল্প:** MapReduce ডিস্কে বারবার লেখার কারণে ধীর। **Apache Spark** RAM-এ প্রসেস করে, তাই অনেক দ্রুত। তবে Spark-ও HDFS-এর উপর নির্ভর করে ডেটা পড়তে এবং লিখতে।

---

## ৪. HDFS High Availability (HA)

Hadoop 1.x-এ NameNode ছিল **Single Point of Failure (SPOF)**। একটি NameNode, সেটি ডাউন হলে পুরো ক্লাস্টার বন্ধ।

Hadoop 2.x থেকে **High Availability** আনা হয়েছে:

- **Active NameNode:** সব রিকোয়েস্ট সরাসরি এটিই হ্যান্ডেল করে।
- **Standby NameNode:** সবসময় Active-এর সাথে সিঙ্ক্রোনাইজড থাকে, প্রস্তুত।
- **Quorum Journal Manager (QJM):** একটি শেয়ারড লগ সার্ভার (সাধারণত ৩টি JournalNode) যেখানে Active NameNode সব পরিবর্তন লেখে এবং Standby পড়ে।
- **ZooKeeper + ZKFC (ZooKeeper Failover Controller):** যদি Active NameNode মারা যায়, Zookeeper সেটি ধরে ফেলে এবং ZKFC স্বয়ংক্রিয়ভাবে Standby-কে Active করে দেয় — মাত্র কয়েক সেকেন্ডের মধ্যে।

## ৫. HDFS-এর সীমাবদ্ধতা

| সীমাবদ্ধতা | ব্যাখ্যা |
|---|---|
| **Small Files Problem** | লক্ষ লক্ষ ছোট ফাইল থাকলে NameNode-এর RAM শেষ হয়ে যায়। প্রতিটি ফাইলের Metadata ~150 bytes RAM খায়। |
| **High Latency** | HDFS দ্রুত Random Read-এর জন্য নয়। এটি বড় Sequential Read-এর জন্য অপ্টিমাইজড। |
| **No Random Write** | একবার লেখা শেষে ফাইলের মাঝখানে কিছু এডিট করা যায় না। শুধু নতুন ডেটা পেছনে যোগ (Append) করা সম্ভব। |
| **NameNode Bottleneck** | সমস্ত Metadata একটি মেশিনের RAM-এ থাকে। Hadoop 3.x-এ Multiple NameNode সাপোর্ট আসছে। |

## ৬. Hadoop ইকোসিস্টেম (Ecosystem)

Hadoop শুধু HDFS এবং MapReduce নয়। এর আশেপাশে একটি বিশাল ইকোসিস্টেম তৈরি হয়েছে:

| টুল | কাজ |
|---|---|
| **Apache Spark** | ইন-মেমরি ব্যাচ ও স্ট্রিম প্রসেসিং |
| **Apache Hive** | HDFS-এর উপর SQL চালানো |
| **Apache HBase** | HDFS-এর উপর রিয়েল-টাইম NoSQL ডেটাবেস |
| **Apache Kafka** | রিয়েল-টাইম ডেটা স্ট্রিমিং |
| **Apache Sqoop** | RDBMS ↔ HDFS ডেটা ট্রান্সফার |
| **Apache Flume** | লগ ডেটা HDFS-এ আনার জন্য |
| **Apache Oozie** | Hadoop জবের ওয়ার্কফ্লো শিডিউলার |
| **Apache Zookeeper** | ডিস্ট্রিবিউটেড কোঅর্ডিনেশন সার্ভিস |

## উপসংহার

Hadoop-এর সাফল্যের মূল রহস্য হলো এর **Horizontal Scalability** — মেশিন ক্লাস্টারে যোগ করতে থাকো, ক্যাপাসিটি বাড়তে থাকবে। একটি সাধারণ ল্যাপটপ থেকে শুরু হওয়া ক্লাস্টার হাজারো মেশিনের বিশাল ক্লাস্টারে পরিণত হতে পারে, কোনো কোড পরিবর্তন ছাড়াই।
